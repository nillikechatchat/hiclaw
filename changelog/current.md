# Changelog (Unreleased)

- feat(manager): teach `worker-management` skill to switch a Worker's runtime in place. `update-worker-config.sh` now accepts `--runtime <openclaw|copaw|hermes>`; when set, the script delegates to `hiclaw update worker --runtime ...` (which updates `Worker.spec.runtime`, makes the controller's reconcile destroy the old container and create a new one from the target runtime's image), then polls `hiclaw get workers -o json` until `phase=Running` and the reported runtime matches. `--package-dir` and `--channel-policy` are explicitly rejected when combined with `--runtime` (they belong to in-place edits, which the recreate path would race with). `worker-management/SKILL.md` adds an operation-reference row, a "Switching Runtime" section, and a gotcha noting that container-local ephemeral state (caches, in-flight task progress) is lost while Matrix account / room / gateway consumer / MinIO data / persisted credentials are preserved. Doc-only for `--remote` workers â€” the controller cannot recreate containers on the admin's machine, so the SKILL points the admin at `lifecycle-worker.sh --action delete` + `hiclaw create worker --remote --runtime <NEW>`.
- fix(controller): propagate `runtime` from `workers-registry.json` to per-team-worker entries during the v1.0.9 â†’ CR migration. `Migrator.createTeamCR` previously only copied `name/model/skills/mcpServers/image` into each `TeamWorkerSpec`, so teams that contained `copaw` or `hermes` workers were silently restarted as `openclaw` after upgrade. Standalone Worker CR migration was already correct; this aligns team workers with the same behavior.

- fix(controller,tests): make hermes workers actually join their own Matrix room, and decouple "send a test message" from worker readiness. Three coupled changes:
  - `service.Provisioner.ProvisionWorker` now follows the same pattern as the human controller: right after creating the room, it calls `matrix.JoinRoom` with the worker's freshly issued access token, so room membership no longer depends on whether the runtime's Matrix client auto-accepts invites. CoPaw's `MatrixChannel` does, `hermes-agent`'s native adapter does not, and previously hermes workers (e.g. `test-15-import-worker-zip` on the hermes shard) sat in `invite` forever and timed out with `Worker did not join room within 120s`. `JoinRoom` is idempotent on the homeserver side, so this is a no-op for runtimes that already self-join.
  - `tests/lib/matrix-client.sh::matrix_wait_for_reply` now filters for the newest `m.room.message` from the target user that has a non-null body, instead of just the newest event of any type â€” hermes-agent emits typing-indicator reactions and redaction events around its real reply, which previously masked the response and caused `Worker did not reply within 120s`.
  - New `matrix_send_and_wait_for_reply` helper implements at-least-once delivery: it sends the prompt, polls for a reply, and resends every 30s if the worker hasn't responded. This is necessary because `membership = join` is *not* a sufficient readiness signal â€” CoPaw's first-boot catch-up sync (see `copaw/src/matrix/channel.py::_sync_loop`) intentionally suppresses message callbacks until `next_batch` is persisted, so any message that arrives in that window is silently dropped. With controller-side pre-join, the test would otherwise race ahead and send into exactly that suppressed window. `test-15-import-worker-zip` now uses this helper and reliably passes for openclaw / copaw / hermes workers without relying on runtime-specific readiness markers. ([dc7ac5a](https://github.com/agentscope-ai/HiClaw/commit/dc7ac5aeca1891f7369b69fb30301b523ac32b38))

- fix(hermes): unblock multi-Agent collaboration with three small but coupled changes that the first end-to-end run on the local embedded stack surfaced (Manager dispatched a task to a Hermes Worker, but the Worker's progress and PHASE-done replies never made it back, the room was spammed with onboarding reminders, and every `rm -rf` froze the workflow waiting for a `/approve` from a human that does not exist):
  - **Outbound `m.mentions.user_ids` (MSC3952) in `hermes_matrix/adapter.py`.** The Hermes Worker prompt already instructs the agent to address peers with their full MXID (e.g. `@manager:matrix-local.hiclaw.io`), but the adapter's `send()` shipped only the markdown-rendered `body` / `formatted_body` and never populated the structured `m.mentions` block. We deliberately do *not* rewrite `body` or inject a `matrix.to` pill into `formatted_body` â€” the structured field is the authoritative signal and matches what every hiclaw adapter already checks.
  - **`HERMES_YOLO_MODE=1` default in the entrypoint.** `hermes-agent`'s `tools/approval.py` already auto-bypasses the dangerous-command gate when its terminal sandbox is `docker`/`singularity`/`modal`/`daytona`, but the Worker runs the `local` sandbox *inside* an already-isolated container, so it sees `env_type=local` and posts `âš ď¸Ź Dangerous command requires approval â€” Reply /approve â€¦` to the room. Users who want the gate back can set `HERMES_YOLO_MODE=0` (or unset it) via `openclaw.env.vars`, the `:=` form preserves any external override.
  - **`MATRIX_HOME_CHANNEL=disabled` default in the entrypoint.** `gateway/run.py` posts a one-time `đź“¬ No home channel is set for Matrix â€¦` reminder whenever a session has no history *and* the platform's `<PLATFORM>_HOME_CHANNEL` env var is unset. For a hiclaw Worker there is no /sethome flow (no human owns the Worker's Matrix client) and no cron jobs to deliver, so the reminder is pure noise that fires on every fresh room session. Any non-empty value satisfies the check; `disabled` is self-documenting and again uses `:=` so an explicit override still wins.
